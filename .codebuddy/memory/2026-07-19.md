# 2026-07-19

## 当前进度快照：Wan2.1 Ulysses POST 变体（显存消融已闭环）

### 实验设计（严格两臂，唯一区别是 POST）
- **baseline (`serial`)**：原始 Q/K/V `nn.Linear` → Q/K RMSNorm → 三次同步 `all_to_all_single` → 3D RoPE → **FA4** → 同步 A2A → 完整 `Wo`。**不使用任何 DeepGEMM 融合算子**。
- **variant (`fused_var`)**：PRE / RoPE / attention 与 baseline **共用同一份代码**；只替换 POST 为 `Wo` 列分片 + GEMM-RS（fwd）、AG-GEMM（bwd）。
- 两者 attention **都是 FA4**（用户明确要求，不能用 SDPA）。
- 关键约束：本机 8 卡全部用于 SP（`SP=8, DP=1`）。**不能把同一个 SP group 再当 FSDP group**，否则 baseline Wo 也被切成 1/8，人为抹掉 variant 收益。

### 代码结构（已落地）
- `examples/wan21/sp/base.py`：公共 `NCCLAllToAll` Function + `_pre_forward`（纯 torch A2A）+ `_attn_forward`（FA4），两臂共用。
- `examples/wan21/sp/serial.py`：baseline POST（A2A + 完整 Wo linear）。
- `examples/wan21/sp/fused_variant.py`：variant POST；`Wo_r_local[hidden, hidden/SP]` 注册后注销完整 `model.o.weight`，不再冗余。
- `examples/wan21/autograd_ops.py`：收敛为 **`FusedPostLinearFunction`**（DeepEP 风格），只描述 autograd 数学；workspace 由策略持有、跨层借用、Function 不分配/销毁/清零。
- `deep_gemm/ag_gemm/__init__.py`：新增 `bf16_ag_gemm_nt_with_input(d, a, b, workspace, num_tokens)`，隐藏 `.ag_x/.ag_slots_x` 内部布局，返回已 gather 的 view 供 Wo 梯度复用。

### 关键正确性修复（不是脏 buffer）
- 旧做法在 Function 里逐调用 `barrier+synchronize+buffer.zero_()` 是错的：GEMM-RS NVLink barrier 自复位、AG C++ launcher 自清 `slot_state`，整块清零是无谓 HBM 流量且可能擦 peer signal。
- 真正竞态：AG comm stream 在远端 rank 尚未把 `grad_y` 写进 `local_x` 时提前 pull。
- 初始正确性修复是 staging 后 `device sync → group barrier → device sync`，但它每层排空所有 CUDA/NCCL stream，破坏 DDP overlap。
- 最终修复已改为 `sym_buffer.handle.barrier()`：barrier 与当前 stream 排序，C++ 随后记录 event 让 AG comm stream 等待，因此不需要 host/device 全局 fence。B300、SP=8 实测所有 rank `grad_X rel=0`，`fwd rel≈0.00287`。

### 显存结论（已实测，B300×8、40 attention 层、BF16 weight/grad + FP32 Adam m/v）
| Seq | Baseline | Variant | 节省 | Workspace |
|---|---:|---:|---:|---:|
| 8K | 48,191.3 MB | 38,277.4 MB | **9,913.9 MB / 20.6%** | 160 MB |
| 32K | 68,686.5 MB | 59,056.6 MB | **9,629.9 MB / 14.0%** | 640 MB |

- 每层完整 Wo ≈ 50 MiB；variant 每卡省 7/8，按 weight+grad+两套 FP32 Adam = 6 倍，40 层理论省约 10.5 GiB，足以覆盖 unified workspace。
- 32K 百分比低于 8K：attention activation 随序列增长，而 Wo 状态节省固定。

### 真实 Wan2.1 14B 权重 + 训练吞吐（2026-07-19 晚）
- 新增 `examples/wan21/checkpoint.py`：读取官方 `diffusion_pytorch_model.safetensors.index.json`，逐 tensor 流式加载，missing key/shape mismatch 直接失败；不再硬编码六分片或使用 `strict=False`。
- 官方 checkpoint 已下载到 HF cache（revision `a064a6c...`）。完整模型用 BF16 参数实测严格加载 **1095 tensors / 14.288B params**，完整 4096-token forward PASS，输出 mean=0.072576/std=0.133118/norm=77.6277。
- 新增 `SPWanTransformer`：40 个官方 T2V block（self-attn + cross-attn + FFN + modulation），仅 self-attn POST 在两臂间变化；训练核心严格加载 **1080 tensors / 14.056B params**。patch/text/time embedding、head 和 optimizer step 不计入吞吐。
- 新增权威入口 `examples/ulysses_variant/bench_wan21_14b_train.py`，默认真实官方权重，显式 `--synthetic` 才随机；报告 rank-max FWD/BWD/SYNC/总耗时、global tokens/s、峰值显存。
- 最终稳定口径：B300×8、SP=8、seq=8192、40 blocks、BF16、DDP overlap、warmup=3、iters=10，吞吐用同步后的 rank-max wall-clock（CUDA event 与 wall 相差 <0.05ms），且 AG 有发布+消费双 barrier：
  - serial：FWD 94.33ms，BWD 185.77ms，总计 280.07ms，29,249.9 tok/s。
  - fused_var：FWD 92.13ms，BWD 198.46ms，总计 290.57ms，28,193.0 tok/s，即 **-3.61%**。反转执行顺序后为 serial 29,425.2、variant 28,417.5 tok/s（**-3.42%**），结论对运行顺序稳定。
  - 仅发布 barrier 的探索值曾为 -4.33%，但缺消费完成协议，存在下一层覆盖风险，不作为最终结论；旧 host fence 对照为 -7.47%。
- 手动同步分解（warmup=3/iters=5）：serial FWD/BWD/SYNC=88.32/125.97/120.81ms，总 334.56ms；variant=87.89/143.54/110.05ms，总 341.50ms。即 variant 纯 BWD 慢 17.57ms，但因少同步 1.049B 个复制 Wo 参数，SYNC 快 10.76ms，最终只慢 **2.03%**。DDP 会隐藏大部分 baseline 同步，因而 variant 的少通信优势不完全转化为 wall-time，最终差距为 3.61%。
- POST 组件微基准 `bench_wan21_post_bwd.py`（8K/SP8）：baseline 每 rank A2A 单向远端 payload 8.75 MiB，variant AG 为 70 MiB，正好 **8×**。典型结果：baseline POST BWD local-op 约 0.19ms；variant 权重转置约 0.03ms、AG+GEMM 约 0.54ms、dW GEMM 0.07ms、dW 转置 0.03ms，总计约 0.67ms。同形状纯 dX GEMM仅约 0.07ms，慢点明确在 AG+GEMM 通信/调度本体，不在矩阵乘形状。组件和不含复制参数 all-reduce，完整 DDP overlap 以上述 40-block wall-clock 为准。
- 梯度同步结论：
  - baseline 的所有参数（含完整 Wo）都在 SP rank 上复制、各自只有局部 token loss 贡献，必须跨 SP reduce。
  - variant 的 q/k/v、norm、bias、cross-attn、FFN、modulation 等仍复制，必须跨 SP reduce；**只有 `Wo_r_local` 跳过 SP reduce**。原因是每 rank 持有不同 Wo 输入列，AG-GEMM backward 已 gather 全序列 `grad_y`，本地 dW 已完整且不同 rank 的 dW 不是同一逻辑参数。
  - 若未来 DP>1，`Wo_r_local` 仍应在“相同 SP 坐标”的 DP group 同步。普通 DDP 应放 DP group；拿 SP group 直接包 vanilla DDP 会错误归约不同 Wo shard。实验的 DDP 模式显式忽略 `_sp_sharded` Wo。

### BWD 根因与下一步
1. 按单向远端 payload，variant AG / baseline A2A = `SP`：baseline 的 total send 中 1/SP 是 self chunk，而 AG 每 rank 要读取其余 SP-1 个完整 local-grad chunk。SP=8 时为 70 MiB vs 8.75 MiB/rank，即 8×。融合只能隐藏这项通信，不能消除它。
2. 当前 AG launcher 在单个 comm stream 上按 peer/chunk 串行提交 `7×4` 次 `cudaMemcpyAsync`，未并发利用多 peer NVLink；AG kernel 又让未就绪 tile 的 CTA 轮询 `slot_state`，可能占住 SM 形成 head-of-line blocking；最后 current stream 还等待 `comm_done_event`，dW 无法覆盖 AG 尾部。
3. 优化优先级：多 peer/多 stream 或 GPU pull 并发 AG；ready-aware tile 调度避免 CTA 自旋占 SM；调 AG tile/cluster heuristic（当前 `N=640` 仅 320 CTA/160 个 2-CTA cluster，K loop 很长）；之后再考虑分块 dW 复用 gathered grad。
4. 在 32K 和更多 bs/seq 组合补真实 14B 吞吐，并增加 optimizer step 的独立口径。
5. 旧 `bench_wan21_fsdp2.py` 已标记**不可用于显存消融**（SP≠FSDP）；显存对比统一用 `bench_wan21_mem_train.py`。
