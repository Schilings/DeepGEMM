### 趋势观察

1. **Rank 数增加 → 融合 kernel 相对更差**：4 GPU/8 GPU 下融合 kernel 的劣势比 2 GPU 更明显
2. **原因**：symmetric push 通信量随 rank 数线性增长（每个 rank 要 push 给 N-1 个 peer），而 NCCL ring reduce_scatter 通信量恒为 `(N-1)/N × data_size`
3. **唯一例外**：4 GPU + 小 shape (512×1024) BF16 得到 5.89x 加速——推测这个 case 下 NCCL 有异常延迟（可能 NCCL 启动开销或 ring 拓扑初始化慢）

## 分析

### 融合 kernel 有优势的场景

仅在 **2 GPU + 小矩阵（tokens_per_rank ≤ 256）** 时融合 kernel 有正收益：

- BF16: 1.03x – 1.49x
- FP8: 1.05x – 1.56x

**核心原因**：
- 小矩阵时 kernel launch overhead 和 NCCL 初始化延迟占比大
- 融合方案省去一次独立通信调用，将计算与 NVLink push overlap
- 2 GPU 时只需 push 给 1 个 peer，通信量最小

### 融合 kernel 劣势的场景（绝大多数）

| 原因 | 说明 |
|------|------|
| NCCL reduce_scatter 高度优化 | Ring/Tree 算法，只传输 (N-1)/N 数据量，bandwidth-optimal |
| 融合 kernel 通信模式非最优 | Symmetric push：每个 rank 向所有 N-1 个 rank 推送 partial，总通信量 = (N-1) × chunk_size |
| GEMM 主体 occupancy 受限 | 融合 kernel 的 epilogue warps (W4-W7) 占了 128 线程，compute-bound 时不如纯 GEMM |
| NCCL 利用 Copy Engine | 分离方案中 NCCL 可以用 CE 与 SM 计算并行，不占 SM |
| Rank 数扩展性差 | 通信量 O(N)，NCCL 通信量 O(1)（对每个 rank 而言） |

### FP32 通信额外开销

FP32 通信在所有配置下都比 BF16 慢 40–100%（NVLink 传输数据量翻倍）。

## 结论与建议

| 场景 | 推荐方案 | 预期收益 |
|------|---------|---------|
| 2 GPU + MoE inference（M_per_rank ≤ 256） | ✅ `gemm_rs_nt` 融合 | 1.3–1.6x |
| 4 GPU + 极小 batch（M_per_rank ≤ 256, N/K 小） | ⚠️ 需实测 | 0.5–5.9x（不稳定） |
| 4+ GPU + 中大矩阵 | ❌ 用 GEMM + NCCL RS | 分离方案快 3–10x |
| 8 GPU（所有 shape） | ❌ 用 GEMM + NCCL RS | 融合 kernel 全面落后 |

**总结：当前融合 GEMM-RS kernel 仅适用于 2 GPU + 小 batch 的 MoE inference 场景。在更多 rank 或更大矩阵下，NCCL 的 bandwidth-optimal 通信算法远胜 symmetric push。**
